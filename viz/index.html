That's the whole player: nearest word, read the colors, add the arrow, repeat. Below we unpack how a map this simple can hold the skill — and why it beats a map built from meaning.

Part 1 — The Idea

Storing skill as geometry

Normally, a good Wordle bot thinks hard on every turn: it looks at all the words that could still be the answer and calculates which guess narrows things down the most. That calculation is repeated from scratch each move. We asked a different question: what if the skill could be baked in ahead of time, so that during the game the bot does almost nothing?

Embedding = giving every word a position (a set of coordinates), so words become dots on a map. Words that "behave alike" in the game sit close together.
Geometry = the shape of that map — which dots are near which. The geometry is what holds the knowledge.

The whole player is just three steps, repeated up to six times:

  1. Guess the nearest word. The bot keeps a single "belief" point on the map. Whatever real word is closest to that point is its guess.
  2. Read the feedback. Wordle answers with colored tiles (green = right letter, right spot; yellow = right letter, wrong spot; gray = not in the word).
  3. Nudge the belief. Based on the guess and the colors, the belief point is moved by a fixed, pre-computed arrow — then repeat.

The key trick: that nudge is not figured out during the game. It was computed once, in advance, and frozen. While playing, the bot never searches or reasons — it just adds a stored arrow and looks for the nearest dot. If a map this simple can still win, then the skill really does live in the shape of the map, not in clever play-time thinking.

The headline

A map built from feedback structure wins about 66–75% of games. A random map wins only 31%. Same simple rule, very different skill — so the skill genuinely lives in how the words are arranged, not in any clever play.

The surprise: meaning is the wrong kind of similarity

A map built from word meanings (GloVe — words like "happy" and "joyful" sit close) wins just 17% — worse than placing words at random (31%). Why? Wordle cares about letters, not meaning. Synonyms can be close in meaning but spelled completely differently, so a meaning-based map actively points the bot the wrong way. The intuitive guess — "use a meaningful word map" — turns out to be a trap.

Part 3 — Moving Downhill

Proof the belief really homes in

A high win rate is nice, but is the bot actually closing in on the answer, or just getting lucky? We can watch directly. Each turn we measure how far the belief point sits from the true answer's spot on the map. We set turn 1 to 1.0 (full distance) and track it shrinking.

Distance from the belief to the true answer over six guesses. Lower is closer. Only the feedback maps dive toward the answer.

The two feedback maps slide steadily downhill toward the answer — from 1.0 down to about 0.55 — exactly what you want from a player that is honing in:

  • MDS (feedback): 1.00 → 0.83 → 0.68 → 0.58 → 0.53 (steady, monotone)
  • Contrastive (SupCon): 1.00 → 0.84 → 0.68 → 0.61 → 0.59 (steady, monotone)
  • Random geometry: 1.00 → 0.95 → 0.88 → 0.86 → drifts back up to 0.91 (sags, then wanders)
  • Semantic (GloVe): 1.00 → 0.98 → 0.96 (barely moves)

This is the mechanical evidence. The maps that win are the maps where the same frozen nudge reliably pulls the belief closer to the answer every turn. The maps that lose either wander or sit still. The "who homes in" split matches the "who wins" split exactly — so the win rate isn't luck, it's the geometry doing real work.

Part 4 — Search vs. Memory

Trading expensive search for cheap memory

Here's why this matters beyond Wordle. The champion solver wins 99.7% — but it pays for it by running a fresh, costly calculation on every single turn. Our frozen map does the opposite: it pays once, up front, to build the map, and then play time is almost free.

Left: the solver re-searches every move. Right: the geometry agent bakes one lookup table once, then just adds a stored arrow at play time.

Amortize = pay a big cost once up front so that each later use becomes tiny. Like cooking a week of meals on Sunday: lots of work once, then dinner is instant all week.

The expensive part — figuring out where every word should go and how to nudge the belief — gets done one time and frozen into a small lookup table. During a game, the bot just reads "guess + colors → arrow," adds the arrow, and grabs the nearest word. No branching search, no candidate lists, almost no computation.

The takeaway: a frozen geometry recovers roughly three-quarters of full-search skill (75% vs 99.7%) with almost no thinking at play time. A real gap to the perfect solver remains — the frozen map is pointed in the right direction but converges a little too slowly to always finish inside six guesses (it wins 73% by guess 6, but 98% if given 15). Still, it shows that a surprising amount of "intelligence" can be compressed into static structure instead of live computation.

How we kept it honest

It would be easy to sneak in cheating — to let the "frozen" bot secretly think during the game. We deliberately tied its hands and checked the knots:

  • The map is frozen at play time. Word positions are fixed before the game starts and never change while playing.
  • No per-move search. The movement is a single pre-computed arrow looked up from a small table built once in advance — zero learned settings and zero searching during play.
  • No peeking at the answer list. The bot never reads the list of still-possible words or filters candidates while playing. It only ever sees its own position, its guess, and the colored feedback.
  • Enforced by an automated test. A test in the code checks that the play-time controller doesn't even import the modules that compute clue patterns or word similarity — so it cannot cheat, even by accident.

All numbers come from a held-out set of 1,158 secret words the maps were never tuned on, using one identical movement rule across every map.
